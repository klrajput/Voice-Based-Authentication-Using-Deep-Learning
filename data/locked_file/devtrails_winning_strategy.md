## PHASE 2: Automation & Protection (Weeks 3-4)

### What Does 4-Star Look Like?

**Core Features (Must-Have for 4-Star):**

1. **Registration Flow** (15 points)
   - Input: Name, phone, delivery platform (dropdown: Swiggy/Zomato/Dunzo), working zone (autocomplete), vehicle type
   - Backend: Store in PostgreSQL, validate phone (10 digits), assign worker_id
   - ML: Risk profiling using zone + platform + vehicle
   - Output: "Welcome Rajesh! Your risk score: 6.5/10"

2. **Dynamic Premium Calculation** (25 points)
   - Input: Worker profile + zone historical weather data
   - ML Model: XGBoost trained on synthetic data (zone_flood_risk, zone_heat_days, worker_hours_per_week, platform_type)
   - Formula: `base_premium (₹80) × zone_multiplier (0.8-1.2) × worker_multiplier (0.85-1.15)`
   - Output: "Your weekly premium: ₹87"
   - **Critical**: Show breakdown in UI ("Base: ₹80, Zone risk: +₹12, Your profile: -₹5")

3. **Policy Management** (15 points)
   - Purchase flow: Select coverage amount (₹1,500/₹2,000/₹2,500), pay premium (mock payment)
   - Policy card: Coverage, premium, active dates, renewal date
   - Backend: Store policy in DB with status (active/expired/claimed)
   - Auto-renewal toggle (for demo purposes)

4. **Claims Automation** (30 points)
   - **3 Parametric Triggers** (must be visible in demo):
     a. Heavy rain: OpenWeatherMap API, check if precipitation >20mm in past 3 hours in worker's zone
     b. Extreme heat: API check if temp >40°C for >4 consecutive hours
     c. Air quality: AQI API, check if AQI >300 in worker's zone
   
   - **Trigger → Claim Pipeline**:

## 5. TECH STACK DECISIONS

**Backend: FastAPI**
- Why: Async support (handles API polling), auto-generated docs, fast development
- Alternative considered: Flask (simpler but no async)
- Decision: FastAPI wins for scalability story in pitch

**Frontend: Streamlit**
- Why: Rapid prototyping, Python-native (no JS needed), impressive demos in <50 lines
- Alternative considered: React (more professional but 3x dev time)
- Decision: Streamlit for Phases 1-2, optionally upgrade to React in Phase 3 if time permits

**Database: PostgreSQL**
- Why: Production-grade (better for pitch), supports complex queries, free tier on Render
- Alternative considered: SQLite (simpler but not scalable)
- Decision: PostgreSQL for demo credibility

**ML: scikit-learn + XGBoost**
- Why: Team familiarity, fast training, easy to explain to judges
- Models:
  - Premium calculation: XGBoost (regression)
  - Fraud detection: Isolation Forest (anomaly detection)
- Alternative considered: Deep learning (overkill for this problem)

**Deployment: Render (free tier)**
- Why: Free PostgreSQL + app hosting, easy GitHub integration, no credit card
- Alternative: Heroku (more expensive), AWS (overkill)
- Decision: Render for demo, can mention AWS in pitch for "scale"

**APIs:**
- Weather: OpenWeatherMap (free tier, 1000 calls/day)
- Air Quality: IQAir or AQICN (free tier)
- Payment: Razorpay test mode (free, India-specific)

**Version Control: GitHub**
- Public repo (required for submission)
- Branch strategy: Simple (main + dev, no feature branches due to time constraint)

## 7. ML STRATEGY (Simple but Visible)

### Premium Calculation Model:

**Model: XGBoost Regressor**

**Why not simpler (linear regression)?** Judges expect "AI" to mean more than y = mx + c.
**Why not complex (neural network)?** Overkill, hard to explain, training time not justified.

**Features (8 total):**
1. zone_flood_risk (0-1 scale, historical flood frequency)
2. zone_heat_days (count of >40°C days per year)
3. zone_pollution_days (count of AQI >300 days)
4. worker_hours_per_week (40-80 hours)
5. platform (Swiggy=1, Zomato=2, Dunzo=3)
6. vehicle_type (bike=1, scooter=2)
7. worker_age (20-50)
8. worker_experience (months, 1-60)

**Target: weekly_premium (₹60-₹120)**

**Training:**
- Synthetic data: 500 workers × 10 zones = 5000 data points
- Formula to generate synthetic premium: `base (80) × zone_risk_multiplier (0.8-1.2) × worker_multiplier (0.85-1.15) + noise`
- Train/test split: 80/20
- Target accuracy: R² > 0.85

**Visibility in UI:**
```
Your Premium: ₹87/week
├─ Base premium: ₹80
├─ Zone risk (Indiranagar): +₹12 (+15%)
└─ Your profile: -₹5 (-6%)
```

This breakdown is CRITICAL for judges to see AI working.

### Fraud Detection Model:

**Model: Isolation Forest**

**Features (5 total):**
1. claim_frequency (claims per month)
2. claim_amount_avg (average claim size)
3. worker_active_hours (total hours worked)
4. zone_avg_claim_frequency (benchmark)
5. claim_time_pattern (variance in claim submission times)

**Output: anomaly_score (0-1)**
- <0.3: Normal, auto-approve
- 0.3-0.7: Medium risk, flag for review
- >0.7: High risk, reject or manual review

**Visibility in UI:**
```
Claim #128
├─ Amount: ₹750
├─ Trigger: Rain (15-Mar)
├─ Fraud Score: 0.82 (HIGH RISK)
└─ Reason: Claim frequency 3x zone average
```

**Training:**
- Synthetic data: 1000 claims (950 normal, 50 fraudulent)
- Fraudulent patterns: High frequency, GPS mismatches, duplicate timestamps

### What Makes ML "Visible" to Judges:

| Invisible (3-star) | Visible (4-star) |
|-------------------|------------------|
| "We use ML for pricing" | "XGBoost model trained on 8 features, R² = 0.87" |

| Fraud detection mentioned | Fraud score shown with reason ("GPS mismatch") |
| Claims processed | Claims dashboard with fraud flags color-coded |

**Key Principle**: If judges can't SEE the AI working in the demo, it doesn't count.

## 8. FRAUD DETECTION STRATEGY

### 2-3 Checks with Highest ROI:

**1. GPS Verification (MUST HAVE - 40% of fraud value)**

**Logic:**
- Worker registers with home zone (e.g., Indiranagar)
- Claim filed for disruption in zone X
- Check: Is zone X within 5km radius of worker's registered zone?
- If NO: Flag as "Location mismatch"

**Implementation:**
```python
from geopy.distance import geodesic

worker_coords = (12.9716, 77.5946)  # Indiranagar
claim_coords = (12.2958, 76.6394)    # Mysore (fraudulent)
distance = geodesic(worker_coords, claim_coords).km

if distance > 5:
    fraud_flag = True
```

**ROI:** Catches workers claiming disruptions in zones they don't operate in. Easy to implement, visually clear in demo.

---

**2. Duplicate Claim Prevention (MUST HAVE - 30% of fraud value)**

**Logic:**
- Same worker + same disruption type + same date = duplicate
- Auto-reject with clear reason

**Implementation:**
```python
existing_claims = db.query(
    "SELECT * FROM claims WHERE worker_id = ? AND trigger_type = ? AND DATE(claim_date) = DATE(?)",
    (worker_id, trigger_type, claim_date)
)

if existing_claims:
    reject_claim(reason="Duplicate claim for same event")
```

**ROI:** Prevents obvious fraud (submitting same claim twice). Zero-cost implementation.

---

**3. Anomaly Detection (NICE TO HAVE - 30% of fraud value)**

**Logic:**
- ML model (Isolation Forest) flags statistically unusual claim patterns
- Features: claim_frequency, claim_amount, worker_hours

**ROI:** Catches sophisticated fraud (workers gaming the system). Impressive to judges because it's ML-based.

---

### Showstopper if Missing:

- **GPS Verification**: Without this, competitors will point out "anyone can claim rain in any zone." This is the baseline fraud check.
